## 📖 خلاصه اجرایی

**آنست** یک سیستم مدیریت انبار سازمانی تمام‌-stack و آماده تولید است که به طور اختصاصی برای مدیریت و ردیابی موجودی در یک زیرس设施 چند سطحی سخت‌گیرانه مهندسی شده است. این پلتفرم یکپارچگی مطلق داده‌ها را با استفاده از **Event Sourcing** تضمین می‌کند و یک تاریخچه ممیزی تغییرناپذیر از هر حرکت فیزیکی موجودی حفظ می‌کند.

در یک محیط VPS لینوکسی امن از طریق Docker containers مستقر شده و از Nginx reverse proxy با خاتمه خودکار SSL، جلوگیری فعال از Brute-force و مانیتورینگ سلامت پس‌زمینه غیرمجاز استفاده می‌کند.

## 🚀 مسئله کسب‌وکار در مقابل راه‌حل مهندسی

### پیچیدگی‌های حوزه

گروه نظارت تاسیسات راه آهن مشهد به یک مکانیزم موجودی خطاپذیر در سایت‌های فیزیکی نقشه‌شده در یک سلسله‌مراتب سخت‌گیرانه (**مرکزی → منطقه‌ای → محلی**) نیاز داشت. معماری‌های CRUD آماده به دلیل چندین واقعیت تجاری سخت‌گیرانه رد شدند:

1. **کنترل جریان جهت‌دار:** تخصیص رو به پایین استاندارد است، اما انتقالات موجودی رو به بالا (مثلاً یک سایت محلی که موجودی را به مرکزی برمی‌گرداند) باید کاملاً غیرممکن باشد.
2. **نشت دفتر مصرف:** ردیابی کالاهای مصرف شده یا فرسوده بدون شکستن حلقه بسته ریاضیاتی سیستم دفتر ۳ سطحی.
3. **عدم انکار مطلق:** مسئولیت کامل حضانت مادی فیزیکی — دانستن دقیق *چه کسی* یک جهش را مجاز کرده، *چه* سریال‌های خاصی جابجا شده و *چرا*.
4. **تحلیل‌های غیرمسدودکننده:** تولید گزارش‌های تاریخی هزینه، نرخ سوخت و مالی بدون قفل کردن جداول تراکنشی داغ عملیاتی.

### نقشه راه معماری

برای ساخت سیستمی مقاوم در برابر وضعیت‌های نامعتبر، تغییر وضعیت را از خواندن وضعیت با استفاده از یک **لایه موتور Event-Sourced** سفارشی جدا کردم.

به جای جهش یک ستون `quantity` سنتی، وضعیت اصلی موجودی به عنوان یک پروژکشن ریاضی مشتق‌شده در نظر گرفته می‌شود. موجودی‌ها به صورت پویا با بازپخش بلوک‌های تاریخی اتمی متوالی (`StockIncreased`, `StockDecreased`) محاسبه می‌شوند.

برای حفظ قوانین سلسله‌مراتب انتقال ۳ سطحی، یک لایه ۴ مجازی معرفی کردم: **«دورریختن»**. کل منطق تجاری و مرزهای فضایی مستقیماً به موتور رابطه‌ای اصلی از طریق تریگرها و رویه‌های ذخیره شده سفارشی منتقل شد.

### تفکیک زیرسیستم‌های جزئی

#### ۱. کلاینت موبایل (Flutter & Dart)

* **ایزولاسیون وضعیت:** با `flutter_bloc` پیاده‌سازی شده تا رویدادهای جریان خام را به مدل‌های وضعیت غیرقابل تغییر متمایز تبدیل کند و مؤلفه‌های بصری را از به‌روزرسانی‌های لایه واکنشی جدا کند.
* **پایپ‌لاین شبکه:** یک نمونه مقاوم `Dio` با اینترسپتورهای خودکار برای پردازش تغییرات هدر HTTP لحظه‌ای، توکن‌های تازه‌سازی JWT و بستن‌های خودکارجلسه `401 Unauthorized` پیکربندی شده.

#### ۲. موتور بک‌اند (.NET 8 Clean Architecture)

* **لایه Domain:** مدل‌های خالص و مستقل از فریمورک حاوی اشیاء تجاری، state‌های enum و مرزهای aggregate بنیادی.
* **لایه Application:** جریان‌های کاربردی ساختاریافته با انتزاع‌های query، مدیریت اعتبارسنجی درخواست ورودی، تبدیل‌های داده و ردیابی وابستگی‌های وضعیت کاربر.
* **لایه‌های Infrastructure & API:** الگوهای کامپایل سریع در ASP.NET Core برای مدیریت پایپ‌لاین‌های ترافیک ناهمزمان با استفاده از EF Core 8 به عنوان انتزاع ORM.

#### ۳. لایه ذخیره‌سازی رابطه‌ای (PostgreSQL 16)

* موتور بهینه‌شده با تریگرهای `PL/pgSQL` برای محاسبات ناهمزمان زیر-میلی‌ثانیه، نماهای ردیابی ساختاری و عملیات کوئری بهینه JSONB.

## 💡 دستاوردهای کلیدی مهندسی

### ۱. پیاده‌سازی ترکیبی Event Sourcing

برای حذف هرگونه خطر جعل ممیزی، جهش‌ها از طریق یک پایپ‌لاین تراکنش append-only متعهد می‌شوند.

* **بازآبیاری وضعیت:** هنگام محاسبه موجودی فعال، برنامه ورودی‌های وضعیت را از جدول `events` می‌کشد.
* **موتور اسنپ‌شات خودکار:** برای جلوگیری از کاهش سرعت اجرا با افزایش حجم رویدادها، یک حلقه `PL/pgSQL` پس‌زمینه لاگ‌های رویداد فعلی را رصد می‌کند. پس از هر ۱۰۰ ورودی به ازای هر aggregate ساختاری، یک اسنپ‌شات کامپایل و در فیلد JSONB `snapshots` ذخیره می‌شود.
* **دریاچه‌های داده مبتنی بر Cron:** یک ابزار محلی موتور رویداد را روزانه کوئری کرده و انتقالات تکمیل شده را به فروشگاه‌های متنی JSON فشرده‌شده صادر می‌کند.

### ۲. محدودیت‌های پایین‌سطح دیتابیس و حفاظ‌ها

تکیه کامل بر سرور وب برای اعتبارسنجی تجاری یک نقطه شکست ایجاد می‌کند. برای جلوگیری از این، منطق سلسله‌مراتب انتقال اصلی مستقیماً در تریگرهای PostgreSQL ساخته شد:

* **اعتبارسنجی توپولوژی:** یک موتور محدودیت `BEFORE INSERT` خط سیر ساختاری را تأیید می‌کند.
* **اجبار جهت جریان:** یک تریگر سفارشی بردارهای انتقال را اعتبارسنجی کرده و هر عملیاتی که سعی در ارسال مواد رو به بالا دارد را متوقف می‌کند.
* **محدودیت‌های ماشین وضعیت:** تغییرات داده به چرخه حیات وضعیت غیرقابل رد شدن (`pending` → `approved` → `in_transit` → `completed`) محدود شده‌اند.

### ۳. کاهش Race Condition همزمانی بالا

```text
عملیات‌های همزمان درخواست حرکت موجودی از یک گره:

کاربر X: درخواست انتقال ──┐  (قفل pg_advisory_xact_lock) ──> [با ایمنی اجرا می‌شود]
                           ├─> [Race Condition مسدود شد]
کاربر Y: درخواست انتقال ──┘  (در صف از طریق قفل تراکنش) ──> [مقدار واقعی را ارزیابی می‌کند]
```

* **راه‌حل:** منطق در یک رویه ذخیره شده منزوی (`execute_stock_transfer`) بر پایه `pg_advisory_xact_lock` محصور شده.
* با ایجاد یک هش قفل صریح متصل مستقیماً به کلید رشته منحصربفرد `WarehouseId + ProductId`، درخواست‌های موازی روی منابع یکسان به ترتیب در صف قرار می‌گیرند.

### ۴. RBAC مبتنی بر منبع آگاه از زمینه

* **دامنه مدیریتی:** مالکان و مدیران عملیات سراسری دسترسی به خواندن، به‌روزرسانی یا دستکاری فیلدهای موجودی در هر گره را دارند.
* **دامنه انباردار:** اپراتورهای استاندارد تسهیلات به وضوح محدود شده‌اند. موتور احراز هویت، JWT ID کاربر فراخوانی کننده را با ورودی `owner_id` اختصاص یافته در جدول `warehouses` مقایسه می‌کند. هر تلاش غیرمجاز فوراً پاسخ `403 Forbidden` را م_trigger می‌کند.

## 🔒 وضعیت امنیتی و DevOps امن‌شده

```text
ترافیک وب عمومی ──[پورت 443]──> [ پروکسی معکوس NGINX ] ──[فقط داخلی]──> [ API وب .NET ]
                                         │                                      │
                                   [ Fail2Ban ] <── (لاگ‌های احراز هویت)    [ PostgreSQL ]
```

* **سخت‌سازی لبه شبکه:** سرور تمام اتصالات عمومی به سمت دیتابیس‌های داخلی را حذف می‌کند.
* **سطح سیستم‌عامل:** احراز هویت رمز عبور SSH کاملاً غیرفعال. دسترسی فقط از طریق کلیدهای `ED25519`. دیمون `Fail2Ban` فعال.
* **پایپ‌لاین‌های اتوماسیون اضافی:** `pg_dump` فشرده هر شب اجرا می‌شود. ۷ روز نگهداری بکاپ.
* **بازرسی‌های سلامت غیرمجاز:** اسکریپت مانیتورینگ هر ۵ دقیقه API را بررسی کرده و در صورت خرابی از طریق `msmtp` ایمیل هشدار ارسال می‌کند.
